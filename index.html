Clean Code

Duration: 1 day

Agenda

  • What is Clean Code
  • Names

Description

The primary value of software is to be flexible. The secondary value of software is to do what business wants it to do. Sounds insane? Just think a bit: what's the difference between IT and regular engineering? What pushed IT so much? The ability to evolve, to adapt to new business needs. And we all know how difficult it is to adapt and strech old, dirty, rigid code.

Financially, the fact that 80% of the costs of a software project are in maintenance may not concern you, the developer, directly. But the fact that you read 10x times more time than you write code should make it clear to you: do your very best to write clean, beautiful code that is easy to maintain. This module explains how to such art.

We will also talk about code ethics, team work, code responsibility and cooperation necessary to any professional software engineer.

We will discuss the most important practical aspects from the reference book 'Clean Code' by Robert C. Martin, plus with many-many other sources.

This is the training which is most likely to have a deep impact on your day-to-day work, changing the way you write code.

The second part of the day we will break the class in two and do a Coding Dojo with each half. During this exercise, we will take turns to write code on the same codebase, discussing design alternatives, tradeoffs, best practices, etc.

1h talk, 1h hands-on (Bowling game kata)

What if instead of writing some tests after writing the actual development, you would begin developing a new feature by first writing a unit test for it. The test will fail!, you would say. It's meaningless!!

Not at all!

Because as soon as you'll see the test passing, that feature is secured from then on. If you do it correctly, you will get 100% real test coverage!.

And what does that mean? It means that you will not fear your own code anymore. You will be able to refactor it as you like, without the terror that you'll break some feature. And as soon as you will refactor it freely, the code will stay alive, it will be able to follow the business requirements closely and cleanly.

Besides the TDD phylosophy and implementation, this module also provides a remarkable series of guidelines for writing good unit tests.

Basic knowledge of unit testing and Java test mocking frameworks (EasyMock, jMock, Mockito,...) is recommended. The JUnit Testing module can bridge that gap for you.

A diagram is the most efficient way to communicate, understand and reason about a software design. That’s why any professional application developer should talk UML fluently. This training covers the UML diagrams most frequently used in the daily practice of a software developer, with particular emphasis on Class, Sequence and Activity Diagrams. Other less common design-time diagrams are also covered, although without going into unnecessary details.

For each exercise, the classroom will be split in teams of 2-3 members, and each team will ellaborate a diagram for each exercise. After the time elapses, each team in turn will explain their solution to the entire classroom to provide a starting point for in-class brainstorming. Alternative modeling options and various design assumptions will be analyzed.

Ok, so you have this huge project written with plain old JDBC code (PreparedStatement ps =..., ResultSet rs =..., } finally { and so on. XML is everywhere.

But do you know how to define the structure of such an XML? And then, how do you work with XMLs in Java?

The XML Schema Definition specification is remarkably complex, because it needs to be as flexible as possible. For regular enterprise applications however, only a (very) limited subset of all these features are used, for example: complex types, simple types, enumerated values, length/min/max restrictions, regex patterns, etc. This module covers all these common constructs, giving you more than 90% of all the XSD knowledge you will probably ever need.

Ok, so you defined your XML structure (.xsd), what then? How do you read/write such XMLs? Do you struggle with SAX events or you cross your fingers and load the whole DOM in memory? Or maybe you are a litle fancier and you play with StAX? All of these standards have their use cases, but for regular enterprise applications, you would probably like to work with JAXB objects that closely resemble your Java Model than with Nodes, Attributes or Events. Let's see how this JAXB works and perhaps how can we customize it a bit.

Enterprise Applications are functionally complex by excellence.

Many times, the architecture of an enterprise application is established beforehand, by some ARCHITECTS, leaving the development team to follow a strategy they don’t fully understand. At the initial design, many enterprise applications are anticipated to grow to a considerable functional complexity, so more elaborate design strategies are often included from the very beginning of the development (=> over-engineer for the early phases). Even so, sometimes the business needs shift to other unanticipated directions, so the architecture needs to be adjusted, maybe even dramatically (=> late redesign). Moreover, even if everything goes as planned, it’s virtually impossible to anticipate all the technical and functional challenges that will be faced by the development team. So if the development team lacks refactoring skills and doesn’t understand the principles and reasons underlying the existing architecture, they will have troubles coping with the various challenges they will face.

Therefore, the development team must learn enterprise architecture principles and common patterns.

But these are hard.

How to teach enterprise design patterns?

How I do it: I start from a trivial “Java-main" application, and gradually add various concerns (such as persistence, user-interface, external services, etc.), asking the participants to brainstorm a solution together. During the training, we will be progressively sketching on a board our current design, so that at the end of the 2 hours, we end up with a Clean Architecture based on the Dependency Inversion Principle, following a pragmatic approach suggested the Adam Bien in Real-World Java EE Patterns (part of the curricula for the Java EE Architect OCM exam).

We won’t see any code, all the discussion is at conceptual level. Therefore, this module is maybe a bit more advanced than the rest. However, it can be discussed in whatever language – the principles remain the same.

TDD, Refactoring

Duration: 1,5 hours / 6-8 devs

How can one grow his coding skills?

Developers working on medium-large applications often don't have the courage/time to refactor, nor the habbit to introspect the design of their code. Too often, it's so much existing code, at it appears to have such a huge inertia that any major improvement feels like a dream. But, as we all know, the moment you stop refactoring and evolving the existing design, at that moment you start writing legacy code. And you don't want to go there.

Even more, you don't want to leave a badly designed legacy of untested code. However bad the design is, a strong suite of tests makes it a lot easier to refactor the code, as you don't fear at any step you take. So you want to start using Test-Driven Development (TDD), one of the most professional methodologies in software development. If you apply it correctly, the code that TDD produces is not only reliably unit-tested but also very-very well decoupled. But you never TDD-ed before.

So you want to learn how to refactor and how to TDD.

As any practices (not frameworks), the key to mastering these is deliberate practice. Just like in Martial Arts, you need to carefully analyze all the moves you do in code. You need to perform Coding Katas: programming exercises in which you don't rush to solve the problem, but intentionally move slower to carefully study all the implications of the decisions you take, with a lot of care for perfection. You will continuously seek to enforce key programming principle as Clean Code, Don't Repeat Yourself (DRY), Keep It Short and Simple (KISS), Single Reponsibility Principle (SRP), as well as learn the refactoring keyboard shortcuts of your IDE. And all that while practicing Test-Driven-Development.

These exercises can be performed individually, but it's way more fun to do them collaboratively. This way we can see how our colleagues think, we can analyze alternative design ideas, discuss tradeoffs and learn handy refactoring shortcuts in a safe 'workshop' athmosphere. When a bunch of people gather together to work on a Kata in a room, it's called a Coding Dojo.

When 4-8 people work together on a single codebase projected on the wall, and they take (e.g. 7 minutes) turns at writing code, it's called a "Randori" format. And why not have a Pair of developers write code together, and then switch the pair when the time's up.

A lot of consultants say that the key to most of the development issues is Pair Programming: one developer writes while another one advises him/discuss together. It disseminates the good technical practices as well as application functional knowledge. It also bounds together the team, etc. etc. And it's proven that Pair Programming is more time-efficient than 2 developers working alone in their caves.

 

A Coding Dojo session consists of 1,5 hours of Randori format. Although I have a preference for Java, we could have it in almost any language of your choice. After several such sessions, the goal is that the trainees to organize other Coding Dojo themselves afterwards, building a technical-centric community.

There are a lot of Katas out there, of various difficulties, each targeting a different kind of codebase and refactoring techniques:

  • Safe Refactoring Kata (existing unit tests)
  • Legacy Code UnitTesting + Refactoring Kata
  • Production-code Kata (later on)

Scala

Duration: 5 days

The course is organized as a continuous workshop: the attendees will write code all the time, as we progress through the topics, to experiment and play with each concept covered. In several such exercises, larger programs will be written. However, due to the extremely compact syntax of Scala, the actual size of the code to write will never represent an impediment – the programs are short, though at times cryptic at first.
